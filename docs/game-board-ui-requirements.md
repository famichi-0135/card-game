# 対戦画面 UI 要件

## 1. 目的

`/games/:gameId` に、2 人対戦を操作するデスクトップ向けのゲームボードを提供する。

- モック `C:/Users/tomop/program/disastarGameNewUI` の領域配置を採用する。
- `@dnd-kit/react` により、手札からのカード操作をドラッグアンドドロップで行う。
- クライアントは公開済みの対戦状態から操作候補を判定するが、ゲーム状態を確定しない。
- 確定状態と最終的なルール判定の正本はバックエンドおよびゲームエンジンとする。

## 2. 対象範囲

### 2.1 対象

- 読み取り専用の対戦状態表示
- 攻撃カードの新規配置、連鎖、手札からの破棄
- サポートカードの対象選択と使用
- 配置フェーズ・サポートフェーズの終了
- コマンド送信中、拒否、再同期、公開イベントの表示

### 2.2 対象外

- モバイル向けの対戦操作レイアウト
- WebSocket を必須とするリアルタイム配信
- 正式なカードイラスト、音声、派手な常時演出
- 観戦、リプレイ、対戦チャット

初期実装は HTTP のスナップショット取得とコマンド送信で動作させる。WebSocket を追加しても、本書の操作・状態・イベントの意味は変更しない。

## 3. 表示環境とレイアウト

- 対戦画面の最小対応表示領域は `1180 x 720` とする。
- これ未満の幅または高さでは盤面を縮小・横スクロールせず、PC の横画面で開くよう案内する。
- キーボード操作は必須とし、ドラッグアンドドロップだけでしか実行できない操作を作らない。

画面は次の 3 領域で構成する。

```text
相手領域: 相手の手札枚数、山札、捨て札、ステータス、みなもと、攻撃グループ 5 枠
中央領域: ラウンド、現在フェーズ、フェーズ残り時間、公開イベント
自分領域: 自分の攻撃グループ 5 枠、みなもと、ステータス、サポート操作、手札、山札、捨て札
```

- 相手の手札は枚数だけを表示する。カード名、画像、順番は表示しない。
- 攻撃グループは自分・相手とも 5 枠を常に表示する。
- 自分の手札は画面下部に並べ、カード詳細を確認できる。
- サポート操作中は、自分側に一時的な操作トレイを表示する。これはゲーム状態のサポートゾーンとは別の UI 状態である。

## 4. カードと攻撃グループの表示

### 4.1 カード

初期実装ではカード画像をプレースホルダーにする。

- カード種別、属性、コスト、攻撃力、カード名、短い効果説明を表示する。
- プレースホルダーは属性・種別を区別できる単純な図形とアイコンで表現する。
- 色だけに依存せず、属性名とアイコンを併記する。
- カード詳細は、ホバー、フォーカス、または選択操作で開く。

現在の公開カードカタログには、効果説明、連鎖条件、画像識別子がない。対戦画面の実装前に、公開用の次のデータを追加する。

```ts
type CardCatalogEntryView = {
  // 既存の公開項目
  id: CardDefinitionId;
  name: string;
  faction: Faction;
  attribute: Attribute;
  cardType: "mana" | "attack" | "support";
  cost: number | null;
  basePower: number | null;

  // UI 用の公開情報
  rulesText: string;
  imageAssetId: string | null;
  interaction: PublicCardInteraction;
};
```

`PublicCardInteraction` は対象数、対象領域、対象所有者などの UI 判定に必要な情報だけを含める。内部 `handlerId`、非公開の効果設定、将来の未公開条件は含めない。

### 4.2 攻撃グループ

- 攻撃グループ枠は各プレイヤー 5 個の固定位置とする。
- 連鎖枚数に上限は設けない。
- 枠内には上から最大 3 枚までを少しずらして重ね、全カード数をバッジで表示する。
- 詳細表示では、グループ内のカードを下から上の順にすべて確認できる。
- 最後に連鎖されたカードを最上段として扱う。

固定位置を維持するため、ゲームエンジンの契約を次のように変更する。

```ts
type AttackGroup = {
  groupId: AttackGroupId;
  slotIndex: 0 | 1 | 2 | 3 | 4;
  // 既存項目
};

type PlaceAttackCardCommand = {
  type: "PLACE_ATTACK_CARD";
  slotIndex: 0 | 1 | 2 | 3 | 4;
  // 既存項目
};
```

`PLACE_ATTACK_CARD` は空の `slotIndex` だけを受理する。グループが除去されても、他のグループの位置を詰めない。`PlayerGameView` には `slotIndex` を公開する。

## 5. 操作仕様

### 5.1 攻撃カード

自分の配置フェーズで、操作可能な攻撃カードだけを選択・ドラッグ可能にする。

| 操作元     | 操作先                 | 送信コマンド        |
| ---------- | ---------------------- | ------------------- |
| 自分の手札 | 空の攻撃グループ枠     | `PLACE_ATTACK_CARD` |
| 自分の手札 | 自分の既存攻撃グループ | `CHAIN_ATTACK_CARD` |
| 自分の手札 | 自分の捨て札           | `DISCARD_HAND_CARD` |

- ドラッグ開始時に、操作可能なドロップ先だけを強調する。
- 無効なドロップ先にはドロップできない。無効理由は選択時・キーボード操作時に表示する。
- 配置、連鎖、破棄は確認ダイアログを出さず、ドロップ直後に送信する。
- カードを場から手札へ戻す、グループ間で移動する操作は提供しない。

### 5.2 サポートカード

サポートカードは、対象選択を完了してから 1 回の `PLAY_SUPPORT_CARD` として送信する。

```text
手札から操作トレイへ置く
  -> 有効な対象を強調する
  -> 必要な対象を選ぶ
  -> 「使用する」で送信
  -> 成功応答の確定状態を表示する
```

- 操作トレイへの移動、対象選択、キャンセルはローカル UI 状態であり、サーバーへ送信しない。
- キャンセル時、カードは確定状態上の手札に残る。
- 対象不要のサポートカードも操作トレイを経由し、「使用する」で確定する。
- 複数対象・複合効果は `PublicCardInteraction` が要求する個数と順序に従って選択する。

### 5.3 フェーズ終了

- `FINISH_PLACEMENT` と `FINISH_SUPPORT` は明示的なボタンで操作する。
- ボタンを押すと、不可逆なフェーズ終了であることを示す確認ダイアログを表示する。
- 承認後にだけコマンドを送信する。
- 自分が操作できないフェーズではボタンを無効化し、理由を表示する。

### 5.4 キーボード操作

- 手札カード、攻撃グループ枠、捨て札、操作トレイ、フェーズ終了ボタンへフォーカス移動できる。
- カードを選択し、対象を選択してから実行できる。
- Escape はドラッグ、対象選択、確認ダイアログを取り消す。
- フォーカス状態と有効・無効状態を視覚的に区別する。

## 6. クライアントの事前判定と送信

Frontend は完全な `GameState` を受け取らず、`executeCommand` を直接呼び出さない。相手の手札や山札を含まない `PlayerGameView` を唯一の表示状態とする。

ゲームエンジンにブラウザ非依存の公開 API を追加する。

```ts
getAvailableGameActions({
  view: PlayerGameView,
  catalog: PublicCardCatalog,
  now: number,
}): AvailableGameActions;
```

この API は、操作可能なカード、カードごとの連鎖先、配置可能な固定スロット、対象候補、無効理由を返す。

- API は公開状態だけで判定できる内容に限定する。
- フェーズ期限、通信遅延、再接続、重複コマンドなどによりサーバーが拒否する可能性は残る。
- D&D の有効化はこの API を使うが、最終判定はバックエンドの `executeCommand` とする。

コマンド送信時は、最新 `PlayerGameView` の `gameId`、`viewerPlayerId`、`phaseSequence`、`stateVersion` を使い、新しい `commandId` を発行する。

## 7. 確定状態、失敗、再同期

- ドロップ直後にカード領域を恒久的に変更しない。ドラッグ中のオーバーレイと、送信中の保留表示だけをローカルに持つ。
- 同一カードまたはフェーズ終了の送信中は、追加のゲームコマンドを送信しない。
- 受理時は応答の `PlayerGameView` を React Query の正規状態として更新し、公開イベントを演出キューへ追加する。
- ルール拒否時は応答に含まれる `PlayerGameView` へ戻し、安定したエラーコードを利用者向け文言へ変換する。
- 通信失敗時は、新しい `commandId` を作らず、同じコマンド ID の結果を確認するためスナップショットを取得する。
- イベント連番が欠落した場合は演出の差分適用を止め、スナップショットで再同期する。
- `prefers-reduced-motion` では、ドラッグ・カード移動・イベント演出を短縮または停止する。確定状態の反映を演出完了まで待たせない。

## 8. 最低限の視覚設計

- 装飾は最小限にし、盤面の情報密度と操作可否を優先する。
- 災害側と対策側は色・アイコン・名称で区別するが、派手な背景や常時アニメーションは使わない。
- カードの種類、属性、コスト、攻撃力、選択状態、送信中、無効状態を一目で判別できるようにする。
- 相手側の非公開情報を推測させる表示、空き枠の存在から非公開情報を示す表示は作らない。

## 9. 実装の分割

1. エンジン契約: `slotIndex`、公開カード操作情報、`getAvailableGameActions`、単体テスト
2. 読み取り専用ボード: 固定 5 枠、カード詳細、プレースホルダー、PC 未対応サイズ表示
3. 攻撃カード操作: D&D、キーボード代替、送信中・拒否・再同期
4. サポート操作: 操作トレイ、対象選択、確認、複数対象
5. 公開イベント演出と WebSocket 対応

## 10. 受け入れ条件

- 1280 x 720 と 1440 x 900 で、盤面、カード、操作要素が重ならず表示される。
- 5 つの固定スロットに新規配置でき、既存グループは削除後も位置を維持する。
- 連鎖カードが 4 枚以上でも、枚数と全カード一覧を確認できる。
- 無効なカードまたはドロップ先から、ゲームコマンドを送信できない。
- サーバーの拒否・通信失敗・イベント欠落後に、最新スナップショットへ復帰できる。
- 相手の手札内容、山札内容・順番、内部カード効果設定が DOM、API レスポンス、ログ表示に含まれない。
- マウス、キーボードだけで、配置・連鎖・破棄・サポート使用・フェーズ終了を完了できる。
