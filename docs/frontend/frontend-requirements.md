# フロントエンド要件定義書

## 1. 文書概要

本書は、Disastar Card Game のフロントエンドが提供する画面、操作、状態管理、品質条件を定義する。

本サービスは、災害側と対策側に分かれて対戦する教育用カードゲームである。フロントエンドでは対戦操作を最優先し、防災情報の閲覧、対戦部屋の作成・参加、認証をその周辺機能として提供する。

本書の優先度は次のとおりとする。

| 優先度 | 意味                                             |
| ------ | ------------------------------------------------ |
| P0     | 対戦を成立させるため、初回リリースに必須         |
| P1     | サービスとして利用するため、初回リリースに含める |
| P2     | 初回リリース後に追加可能                         |

ゲームルールの正本は[基本ゲームルール定義書](../game/rule-definition.md)、ゲーム状態とコマンドの正本は[ゲームエンジン仕様書](../game/gameEngine-definition.md)、通信境界の正本は[クライアント・サーバー通信境界](../backend/client-server-protocol.md)とする。本書はそれらをUIへ反映する方法を定義し、ルールそのものを再定義しない。

### 1.1 対戦画面の優先順位

[対戦画面 UI 要件](./game-board-ui-requirements.md)は、盤面、カード操作、公開情報、再同期の詳細を定義する。本書と同要件が矛盾する場合は、対戦画面 UI 要件を優先する。

本書では、次の相違を解消済みとする。

| 論点           | 採用する仕様                                                                                                                  |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 対象画面       | 対戦は最小 `1180 x 720` の PC 横画面だけを対象にする。横向きタブレットの専用対応は初期実装に含めない。                        |
| 攻撃グループ   | 各プレイヤーに `slotIndex` を持つ 5 個の固定枠を常時表示し、除去後も他の枠を詰めない。                                        |
| 確認操作       | 配置、連鎖、破棄は即時送信し、確認ダイアログはフェーズ終了だけに表示する。                                                    |
| サポートカード | 手札からローカルの操作トレイへ置き、対象選択後に「使用する」で 1 回だけ送信する。                                             |
| 通信           | コマンド送信と正規状態取得はHTTPで行い、WebSocketは更新通知と一時的なプレゼンスを配信する。切断時はHTTPポーリングで復旧する。 |

実装の契約ゲート、責務ごとの変更判断、PR 分割は[フロントエンド連携の実装計画](./frontend-integration-plan.md)を正とする。対戦画面の実 API 接続は、同計画のフェーズ 0 を完了してから行う。

## 2. 確定事項

- 対戦画面は最小`1180 x 720`のPC横画面を対象とする。
- スマートフォンには対応しない。
- 対戦画面は、ページを縦スクロールせず1画面でゲームを完結できる構成とする。
- カード操作には`dnd-kit`を使用する。
- 攻撃カードの配置・連鎖、サポートカードの使用、手札の破棄をドラッグ＆ドロップで操作できるようにする。
- 配置が確定したカードは、通常操作では移動できない。
- サポートカードは操作トレイに置いてから対象を選択し、「使用する」で確定する。
- 配置、連鎖、破棄は確認せず送信し、確認ダイアログはフェーズ終了だけに表示する。
- 初期実装のカード画像はプレースホルダーとする。
- フロントエンドでも共通ゲームロジックを使用して操作を事前検証する。
- 最終的な操作可否と状態確定は常にバックエンドが行う。
- タブ非表示中はWebSocket接続を休止し、表示復帰時に再接続する。
- トップページと防災情報は未ログインでも閲覧できる。
- 部屋作成、部屋参加、対戦はログインを必須とする。
- 防災情報は初期段階ではリポジトリ内のMarkdownまたはMDXで管理する。

## 3. 対象利用者

- 災害や防災を学びながらカードゲームを遊ぶ一般利用者
- 学校やイベントなどで本サービスを利用する学習者と進行担当者
- 防災情報だけを閲覧する未ログイン利用者

初回リリースでは、観戦者、管理者、記事編集者向けの専用画面は対象外とする。

## 4. 対応環境

### 4.1 対象画面

| 環境                             | 対応方針                                                             |
| -------------------------------- | -------------------------------------------------------------------- |
| PC 横画面 `1180 x 720` 以上      | 対戦に必要な全領域と操作を1画面に表示する。                          |
| 幅または高さが最小値未満         | 盤面を縮小・横スクロールせず、対応する PC 横画面で開くよう案内する。 |
| スマートフォン・横向きタブレット | 初期実装では対戦UIを提供しない。                                     |

トップページ、防災情報、認証画面もスマートフォン最適化は初期要件に含めない。ただし、対戦画面以外では対応対象の PC 横画面で内容や操作が欠落しないことを必須とする。

### 4.2 対象ブラウザ

初回リリース時点の最新版および1つ前の主要バージョンを対象とする。

- Google Chrome
- Microsoft Edge
- Safari（macOS）

ドラッグ＆ドロップ、HTTP通信、CSS Gridなど、利用機能の実動作を各対象ブラウザで確認する。WebSocketを導入した段階で、同じ対象ブラウザで追加確認する。

## 6. カードゲーム対戦画面

### 6.1 目的

対戦中に必要な公開情報、カード操作、フェーズ操作、通信状態を1画面に集約し、ルールを知らない利用者でも「現在できる操作」を判断できるようにする。

### 6.2 レイアウト

初期のレイアウト参照として、[盤面モック](./card-game-ui-image/image.png)を使用する。ただし、この画像は領域配置を示すスケッチであり、カード枚数、固定スロット、操作可否の正本は本書と対戦画面 UI 要件とする。

画面は次の3領域で構成する。

```text
相手領域: 相手の手札枚数、山札、捨て札、ステータス、みなもと、攻撃グループ5枠
中央領域: ラウンド、現在フェーズ、フェーズ残り時間、公開イベント、フェーズ終了操作
自分領域: 自分の攻撃グループ5枠、みなもと、ステータス、サポート操作トレイ、手札、山札、捨て札
```

情報パネルはカードやドロップ領域を覆わない固定レイアウト領域とする。相手の手札は枚数だけを表示し、カード名、画像、順番を表示しない。

#### ゲーム進行バー

常に次を表示する。

- ラウンド番号と最大ラウンド数
- 現在のフェーズ
- 操作中のプレイヤー
- フェーズ残り時間
- 接続状態
- 再同期状態
- メニューまたは退出操作

残り時間は`phaseDeadlineAt`から表示用に計算する。ブラウザの時計による表示は参考値であり、期限判定には使用しない。

#### プレイヤー情報

双方について次を表示する。

- 表示名
- 災害側または対策側の陣営
- スタミナ
- 属性ごとのみなもと総量と使用中量
- 山札残数
- 手札枚数
- 捨て札
- サポート終了状態

相手の手札は枚数とカード裏面だけを表示し、カード定義IDや内容をDOM、ログ、アクセシビリティ名にも含めない。

#### バトルゾーン

- 自分と相手に5つの固定攻撃グループ枠を常時表示する。グループには`slotIndex`を持たせ、除去後も他のグループを移動しない。
- 既存グループには、そのグループを構成するカードを連鎖順が分かる状態で重ねて表示する。上から最大3枚をずらして表示し、4枚以上は総枚数バッジと詳細表示で確認できるようにする。
- グループの属性、必要みなもと、現在の攻撃力を表示する。
- サポートカードの選択中は、自分側に一時的な操作トレイを表示する。操作トレイはゲーム状態のサポートゾーンとは別のローカルUI状態である。
- 相手の領域は閲覧専用とし、効果対象として選択できる場合だけ選択可能状態を表示する。

#### 手札

- 最大5枚を常時識別できる大きさで表示する。
- カード名、画像、種別、属性、コスト、攻撃力または効果概要を表示する。
- ホバー、クリック、キーボードフォーカスで詳細を確認できる。
- カード詳細表示とドラッグ開始を誤認しないよう、ポインター移動量のしきい値を設ける。
- 初期実装では属性・種別を区別できるプレースホルダー画像を使用し、色だけに依存せず属性名とアイコンを併記する。

### 6.3 フェーズ別の操作

#### 攻撃カード配置フェーズ

操作対象プレイヤーは次を実行できる。

| 操作                       | UI操作                               | 送信コマンド        |
| -------------------------- | ------------------------------------ | ------------------- |
| 新しい攻撃グループを作る   | 攻撃カードを空きグループ枠へドロップ | `PLACE_ATTACK_CARD` |
| 既存攻撃グループへ連鎖する | 攻撃カードを対象グループへドロップ   | `CHAIN_ATTACK_CARD` |
| 手札を捨てる               | 手札を自分の捨て札領域へドロップ     | `DISCARD_HAND_CARD` |
| 配置を終了する             | 配置終了ボタン                       | `FINISH_PLACEMENT`  |

- サポートカードとみなもとカードは攻撃グループへドロップできない。
- 既に配置が確定したカードはドラッグできない。
- 捨て札へのドロップ後は確認せず、直ちにコマンドを送信する。
- 配置終了は取り消せないことをボタン付近で明確にする。

#### サポートフェーズ

双方のプレイヤーは次を実行できる。

| 操作                     | UI操作                                                     | 送信コマンド        |
| ------------------------ | ---------------------------------------------------------- | ------------------- |
| サポートカードを使用する | 手札から操作トレイへ置き、対象を選択して「使用する」を押す | `PLAY_SUPPORT_CARD` |
| サポート操作を終了する   | サポート終了ボタンを押し、確認ダイアログで承認する         | `FINISH_SUPPORT`    |

- 操作トレイへの移動、対象選択、キャンセルはサーバーへ送信しない。キャンセル時、カードは確定状態上の手札に残る。
- 対象不要のサポートカードも操作トレイを経由し、「使用する」で確定する。
- 終了宣言済みのプレイヤーは新しいサポートカードをドラッグできない。
- 片方が終了済みでも、もう片方は終了または時間切れまで操作できる。
- 双方がほぼ同時に操作した場合、画面表示ではなくサーバー受信順を正とする。

#### 自動処理フェーズ

ラウンド解決、場の整理、手札補充などの自動処理中はカード操作を無効化する。現在の処理名を表示し、公開イベントを使って結果を演出するが、確定した状態の反映を演出完了まで遅らせない。

### 6.4 ドラッグ＆ドロップ

`dnd-kit`を使用し、次の状態を実装する。

1. ドラッグ開始時に、最新の`PlayerGameView`と公開カードカタログから合法な操作を計算する。
2. 合法なドロップ先だけを強調表示する。
3. 不正な領域は無効表示にするが、色だけで可否を伝えず、形状、アイコン、ラベルも併用する。
4. ドラッグ中は元カードの領域を維持し、専用のDrag Overlayを表示する。
5. 攻撃カードの合法なドロップ時は、対応するコマンドを直ちに送信する。破棄にも確認ダイアログを表示しない。
6. サポートカードは操作トレイへ置き、対象選択後に「使用する」でコマンドを送信する。
7. フェーズ終了だけは、確認ダイアログの承認後にコマンドを送信する。
8. コマンド送信中は同じカードまたはフェーズ終了の重複操作を防止する。
9. サーバー受理後に`PlayerGameView`を更新する。
10. サーバー拒否時はレスポンスに含まれる正規状態へ同期し、操作を元に戻して理由を表示する。

フロントエンドだけでゲーム状態を確定する楽観的更新は行わない。ドラッグ中のプレビューと送信中表示だけを一時的なUI状態として保持する。

ドラッグ中にフェーズまたは`phaseSequence`が変わった場合は操作をキャンセルし、最新状態へ同期する。

### 6.5 カード効果の対象選択

対象を必要とするサポートカードを操作トレイへ置いた場合、次の順に処理する。

1. カード定義の効果順に対象選択を開始する。
2. 選択可能なカード、グループ、プレイヤー、みなもとだけを強調する。
3. 必要数、選択済み数、効果概要を表示する。
4. すべての必須対象を選択するまで確定ボタンを無効化する。
5. キャンセル時はカードを手札表示へ戻し、コマンドを送信しない。
6. 確定時に効果IDごとの`EffectInput`を組み立て、カード操作コマンドを送信する。

対象の解決順はクライアントが選んだ順ではなく、カード定義の`effects`順を正とする。

### 6.6 クライアント側ルール検証

フロントエンドから完全な`GameState`や相手の非公開情報を参照してはならない。`@disastar/game-engine`に、ブラウザから安全に利用できるプレイヤー視点の検証APIを追加する。

検証APIは少なくとも次を入力として受け取る。

- `PlayerGameView`
- 公開用カードカタログ
- 操作しようとしているカードインスタンスID
- 操作候補となるドロップ先または効果対象

検証APIは少なくとも次を返す。

- 操作可能か
- 対応するゲームコマンド種別
- 合法なグループID、固定スロット、ゾーン
- 必要な効果入力
- 操作不能理由を表す安定したコード

実装時は、`executeCommand`内の検証と別ルールを複製せず、公開情報だけで判定できる純粋関数を共通化する。相手の同時操作などにより、クライアント検証後でもサーバーが拒否できることを前提とする。

クライアント側検証の目的は、次に限定する。

- 不正なドロップ先を事前に無効化する
- 操作不能理由を即座に表示する
- 不要なAPIリクエストを減らす

サーバー側の`executeCommand`による再検証は省略しない。

### 6.7 コマンド送信

各コマンドには次を設定する。

- 操作ごとに一意な`commandId`
- URLと一致する`gameId`
- 認証中の利用者と一致する`playerId`
- 操作開始時点の`phaseSequence`
- 操作開始時点の`clientStateVersion`
- 表示・診断用の`issuedAt`

同一コマンドの応答待ち中に再操作されても、新しい`commandId`で同じ操作を送信しない。通信タイムアウト後の再送では同じ`commandId`を使用する。

### 6.8 接続・再同期

HTTPのスナップショット取得とコマンド送信を正本とする。WebSocketは更新通知と一時的なプレゼンスを配信する。タブが非表示の間はWebSocket接続を休止し、表示復帰時に再接続する。更新通知を受信した場合も、同じHTTP再同期と公開イベントの意味を使用する。

操作責任を持つプレイヤーがそのフェーズの制限時刻までに接続へ戻らない場合、そのプレイヤーは切断敗北とする。サポートフェーズで未終了の両者が接続していない場合は引き分けとする。

次の接続状態を画面上で区別する。

- 接続中
- 接続済み
- 再接続中
- オフライン
- 再同期中
- 復旧不能

イベント連番の欠落、再接続、サーバー拒否による状態差異を検出した場合は、差分適用を止めてスナップショットを取得する。再同期中はカード操作を無効化する。

HTTP 初期実装では、ページが可視かつゲームが終了していない間に、最後に確認したイベント連番を指定して2秒ごとにスナップショットを取得する。タブが非表示の間は停止し、フォーカス復帰時に直ちに同期する。コマンド POST が通信失敗した場合は、新しい`commandId`を発行せず、同じ本文を同じ`commandId`で再送する。スナップショットは表示状態の再同期専用であり、特定コマンドの受理・拒否を推測してはならない。

接続が切れてもゲームの制限時間は止まらない。復帰後はローカル状態を破棄し、取得した`PlayerGameView`を正とする。

### 6.9 終了状態

ゲーム終了時は次を表示する。

- 勝利、敗北、引き分け
- 終了理由
- 最終ラウンド
- 双方の最終スタミナと総パワー
- トップへ戻る操作
- 同じ相手との再戦導線はP2とする

終了後はカード操作とフェーズ終了操作を無効化する。

## 7. 災害・防災情報画面

### 7.1 目的

ゲームで扱う災害、防災行動、防災サービスについて、根拠となる情報をゲーム外でも学べるようにする。

### 7.2 一覧画面

- 災害情報、防災行動、防災サービスのカテゴリを表示する。
- 記事タイトル、要約、カテゴリ、更新日を一覧表示する。
- カテゴリによる絞り込みを提供する。
- キーワード検索はP2とする。

### 7.3 詳細画面

- 見出し、本文、画像、出典、最終確認日を表示する。
- 外部出典は新しいタブで開き、外部サイトであることを示す。
- 緊急時は自治体、気象庁などの最新情報を確認すべきことを明記する。
- 本サービスの説明をリアルタイムの避難判断に使用しないよう注意を表示する。
- 関連するゲーム内カードへのリンクはP2とする。

### 7.4 コンテンツ管理

初期段階ではMarkdownまたはMDXファイルをGit管理する。各記事は少なくとも次のメタデータを持つ。

- `title`
- `summary`
- `category`
- `tags`
- `sourceName`
- `sourceUrl`
- `reviewedAt`

公的機関などの一次情報を優先し、出典のない断定的な防災情報を掲載しない。Markdown内の任意HTMLは許可せず、レンダリング時に安全な要素だけを許可する。

CMS、利用者による投稿、リアルタイム災害速報は初期要件に含めない。

## 8. トップ・部屋作成参加画面

### 8.1 目的

サービスを開いた利用者が、説明を読み進める前に対戦の作成または参加へ進めるようにする。マーケティング用の大型ヒーローではなく、対戦開始操作を最初の画面の中心に置く。

### 8.2 未ログイン状態

- 部屋IDまたは招待URLの入力欄を表示する。
- 作成または参加を選ぶとログイン画面へ移動する。
- ログイン後に入力済み部屋IDまたは作成操作へ復帰できるようにする。
- 防災情報一覧への導線を表示する。

### 8.3 ログイン済み状態

- 保存済みの30枚デッキから災害側または対策側を選び、部屋を作成できる。
- 部屋IDまたは招待URLを入力し、部屋へ参加できる。
- 入力されたURLから安全に`matchId`を抽出し、他オリジンへの移動には使用しない。
- デッキがない場合は、両陣営のスターターデッキを作成する導線を表示する。
- 最近の対戦履歴はP2とする。

### 8.4 対戦待機部屋

- 部屋IDと招待URLを表示し、コピーできる。
- 作成者の選択陣営を表示する。
- 参加者は作成者と異なる陣営のデッキだけを選べる。
- 作成者は対戦開始前に部屋を取り消せる。
- 対戦成立後は双方を`/games/:gameId`へ移動する。
- 存在しない、取り消し済み、開始済みの部屋を区別して表示する。

## 9. 認証とアカウント操作

Better Authが提供するセッションを使用し、次を実装する。

- メールアドレスとパスワードによる登録
- 確認メールの再送を含むメールアドレス確認
- ログイン
- ログアウト
- パスワード再設定メールの送信
- パスワード再設定
- セッション期限切れ時の再ログイン

認証中、成功、入力エラー、サーバーエラーを区別する。認証エラーでメールアドレスの登録有無を第三者が判別できる文言は表示しない。

ログアウトはヘッダーまたはアカウントメニューから実行し、成功後はトップへ戻す。対戦中のログアウトでは確認を求める。

プロフィール編集、アバター、外部OAuth、二要素認証は初期要件に含めない。

## 10. 共通UI状態

すべての画面で次を考慮する。

- 初回読み込み
- データなし
- 入力不備
- 権限不足
- APIエラー
- ネットワーク切断
- セッション期限切れ
- 処理中の多重操作防止
- 404

エラーコードをそのまま表示せず、利用者が次に取れる行動を日本語で示す。詳細コードは診断用として必要な場合だけ補助表示する。

## 11. アクセシビリティと操作性

スマートフォン非対応でも、ポインター操作だけに限定しない。

- すべての主要操作をキーボードで実行できる。
- カードへフォーカスし、EnterまたはSpaceで持ち上げ、合法な対象へ移動して確定できる。
- Escapeでドラッグまたは効果対象選択をキャンセルできる。
- ドラッグ開始、合法な対象、ドロップ完了、拒否理由を支援技術へ通知する。
- 操作可否や陣営、属性を色だけで区別しない。
- フォーカスリングを常に視認できるようにする。
- アニメーション低減設定を尊重する。
- カード画像にはカード名と役割が分かる代替テキストを付ける。
- カード詳細ポップオーバーは、ホバーだけでなくクリックとキーボードでも開閉できる。

## 12. データと依存境界

依存方向は次を維持する。

```text
@disastar/game-engine  ← プレイヤー視点の合法操作判定を含む純粋ロジック
        ↑
@disastar/contracts    ← 通信DTO
        ↑
apps/frontend          ← 表示、入力、演出、一時的UI状態
```

フロントエンドは`GameState`を保持せず、`PlayerGameView`を正規の表示状態とする。

カード表示には、現在の`PlayerGameView`だけでは不足するため、バックエンドは非公開実装を除いた公開カードカタログを提供する必要がある。公開カードカタログには少なくとも次を含める。

- カード定義ID
- カード名
- 陣営
- 種別
- 属性
- コスト
- 基本攻撃力
- 効果の表示用説明
- 画像アセット識別子
- UI判定に必要な対象数、対象領域、対象所有者、連鎖条件

公開カタログは`PlayerGameView.cardCatalogVersion`を指定して取得する。不変な対戦状態に対して最新カタログを黙って使わない。ブラウザは同一オリジンの`/api/*`だけを呼び出し、認証 Cookie は Better Auth のセッションに任せる。ブラウザから Backend Worker の URL を直接呼んだり、独自トークンを保存したりしない。

公開情報は`PublicCardInteraction`として定義し、カスタム効果の内部ハンドラーID、未公開カード、対戦相手の非公開情報は返さない。

## 13. 非機能要件

### 13.1 表示安定性

- 対戦画面は1180×720、1280×720、1440×900でページ縦スクロールと盤面の横スクロールを発生させない。
- ドラッグ、残り時間更新、カード追加で領域全体の配置をずらさない。
- 攻撃グループ、カード、タイマー、操作ボタンには安定した寸法を与える。
- 最長のカード名や日本語エラー文が隣接領域へ重ならない。

### 13.2 性能

- ドラッグ中にゲーム状態全体を再計算せず、ドラッグ開始時の合法操作結果を再利用する。
- タイマー更新によってカード一覧全体を再描画しない。
- カード画像は表示寸法に合う形式と大きさで配信する。
- 初期表示に不要な防災記事とカード画像は遅延読み込みする。

### 13.3 セキュリティと情報秘匿

- 相手の手札と山札の内容をクライアントへ送らない。
- URLの`gameId`、`matchId`、戻り先を信頼せず検証する。
- Markdownを安全な要素だけに変換し、任意スクリプトを実行しない。
- 認証セッションはBetter Authの既存方針に従い、ブラウザストレージへ独自トークンを保存しない。
- クライアント側バリデーションを認可や不正防止の根拠にしない。

## 14. テスト要件

### 14.1 共通ゲームロジック

- 各フェーズで合法なカードと対象だけを返す。
- みなもと不足、グループ上限、連鎖不可、異なる属性を不正と判定する。
- 配置済みカードを移動可能として返さない。
- 非公開情報がなくても判定可能な範囲だけを扱う。
- クライアント事前判定とサーバー`executeCommand`の結果が同じ入力で矛盾しない。

### 14.2 対戦画面

- 1180×720、1280×720、1440×900で必須領域と主要操作が同時に見える。
- 攻撃カードを空き枠へドロップすると`PLACE_ATTACK_CARD`を送る。
- 合法な既存グループへドロップすると`CHAIN_ATTACK_CARD`を送る。
- サポートカードは操作トレイと対象選択を経て、明示的な「使用する」で`PLAY_SUPPORT_CARD`を送る。
- 捨て札へのドロップ時は確認せず`DISCARD_HAND_CARD`を送る。
- `FINISH_PLACEMENT`と`FINISH_SUPPORT`だけは確認ダイアログの承認後に送る。
- 不正な対象へドロップしてもコマンドを送らない。
- 効果対象をキャンセルした場合にコマンドを送らない。
- コマンド拒否時に正規状態へ戻り、理由を表示する。
- 再接続とイベント欠落時にスナップショットへ再同期する。
- 相手の非公開カード情報がDOMとログに存在しない。
- キーボードだけでカード配置とキャンセルを実行できる。

### 14.3 周辺画面

- 未ログインでもトップと防災記事を閲覧できる。
- 未ログインで部屋参加を選ぶと、ログイン後に元の部屋へ戻る。
- 異なる陣営のデッキで部屋を成立させ、対戦画面へ移動できる。
- 登録、メール確認、ログイン、ログアウト、パスワード再設定の各状態を表示できる。

## 15. 初回リリースの完了条件

次をすべて満たした時点でフロントエンド初回リリースを完了とする。

- ログイン済みの2人が部屋を作成・参加し、異なる陣営で対戦を開始できる。
- カード効果を含む全プレイヤーコマンドを対戦画面から送信できる。
- 合法なD&D対象が事前表示され、不正操作がサーバー送信前に抑止される。
- サーバー拒否、タイムアウト、再接続後も正規状態へ復帰できる。
- 対戦開始から勝敗表示までページ遷移せず進行できる。
- 1180×720、1280×720、1440×900でページ縦スクロールと盤面の横スクロールなしに対戦できる。
- 公開情報と非公開情報の境界を破らない。
- 未ログイン利用者が防災情報を閲覧できる。
- 認証とパスワード再設定の基本フローが動作する。
- 本書のテスト要件をCIで継続的に確認できる。

## 16. 実装順序

実装は次の単位に分け、対戦画面と周辺画面を同じPRへ混在させない。

1. ゲームエンジンとバックエンドの契約ゲート: 固定スロット、公開盤面数値、公開カードカタログ、合法操作判定、バージョン解決
2. Frontend 基盤: ルーティング、React Query、`@dnd-kit/react`、認証ガード、API クライアント
3. `PlayerGameView`から描画する読み取り専用の対戦画面
4. HTTP の差分ポーリング、攻撃配置、連鎖、破棄の D&D 操作
5. サポート使用、効果対象選択、コマンド拒否、再送、再接続、イベント演出
6. 認証画面と戻り先制御
7. トップ、部屋作成・参加、待機部屋
8. MarkdownまたはMDXによる防災情報画面

各段階で、実装対象の受け入れ条件を満たしてから次へ進む。

## 17. 対象外

- スマートフォン向け対戦UI
- 観戦
- ランクマッチと自動マッチング
- チャット
- デッキ編集画面
- 対戦リプレイ
- CMS
- 利用者による防災記事投稿
- リアルタイム災害速報
- 外部OAuthと二要素認証

対象外機能を追加する場合は、既存の公開情報境界と対戦画面の1画面要件を維持できるか再検討する。
