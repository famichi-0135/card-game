# 基本ゲームルール定義書

## 1. 文書概要

本書は、2人対戦型カードゲームにおける基本ゲームルールを定義する。

ゲームの振る舞いに関する正本は本書とする。状態、コマンド、イベントなどの技術契約は「ゲームエンジン仕様書」、カードごとの効果定義と追加規則は「カード効果仕様書」で管理する。

実装時の共有型は`packages/game-engine/src/contracts/`を唯一の正本とし、文書ごとに互換性のない型を作らない。

---

## 2. 基本用語

### 2.1 ゲーム

2人のプレイヤーがカードを使用して総パワーを競い、相手のスタミナを減らす対戦を指す。

### 2.2 ラウンド

先攻プレイヤーの配置フェーズから、次のラウンド開始までの一連の処理を指す。

### 2.3 フェーズ

ラウンド内の処理区分を指す。

1ラウンドは、次のフェーズによって構成される。

1. 先攻プレイヤー配置フェーズ
2. 後攻プレイヤー配置フェーズ
3. サポートフェーズ
4. スコア計算フェーズ
5. 勝敗判定
6. 場の整理
7. 山札切れ事前判定
8. 手札補充フェーズ
9. 次ラウンド開始

ゲームエンジンは各フェーズの実体へ単調増加する`phaseSequence`を付与し、別ラウンドの同名フェーズを区別する。

### 2.4 みなもと

攻撃カードおよびサポートカードをバトルゾーンに維持するために必要な、属性ごとの最大許容コストを指す。

みなもとは消費して失われるリソースではない。現在バトルゾーンに存在するカードが必要とするコストによって、一時的に使用可能量が減少する。

### 2.5 攻撃グループ

バトルゾーンに配置された1枚以上の攻撃カードのまとまりを指す。

連鎖された攻撃カードは、1つの攻撃グループとして扱う。

### 2.6 陣営

各プレイヤーが担当する教育上の役割を指す。陣営は`disaster`（災害・環境破壊側）と`countermeasure`（防災・環境対策側）の2種類とする。

1ゲームには各陣営のプレイヤーが1人ずつ参加する。同じ陣営同士ではゲームを開始しない。勝敗処理はプレイヤー単位で行い、陣営による特別な勝敗補正は設けない。

---

## 3. ゲームの基本条件

| 項目             | 内容                      |
| ---------------- | ------------------------- |
| プレイヤー数     | 2人                       |
| 陣営             | 災害側1人・対策側1人      |
| デッキ枚数       | プレイヤーごとに30枚      |
| 初期スタミナ     | 25                        |
| 初期手札         | 5枚                       |
| 手札上限         | 5枚                       |
| 最大ラウンド数   | 30ラウンド                |
| 攻撃グループ上限 | プレイヤーごとに5グループ |
| 攻撃配置時間     | プレイヤーごとに最大90秒  |
| サポート時間     | フェーズ全体で最大60秒    |

攻撃配置時間およびサポート時間は、今後のゲームバランス調整によって変更される可能性がある。

---

## 4. カードの種類

カードは、次の3種類に分類する。

### 4.1 みなもとカード

みなもとの総量を増加させるカード。

- みなもとカード1枚につき、対応する属性のみなもと総量を1増加させる
- 引かれたみなもとカードは手札に残らない
- 効果を処理したみなもとカードは捨て札に移動する
- みなもとカードをプレイヤー自身の操作で捨てることはできない
- みなもとカードはバトルゾーンにカード実体として残らない
- バトルゾーンには、属性ごとのみなもと数値だけが記録される

### 4.2 攻撃カード

バトルゾーンの攻撃グループに配置され、総パワーの計算に使用されるカード。

攻撃カードは、少なくとも次の情報を持つ。

- カード名
- 属性
- コスト
- 基礎攻撃力
- 上に連鎖できるカードの情報
- 効果配列

将来の攻撃カード効果追加で型と配置処理を作り直さないよう、攻撃カードは契約上`effects`を持つ。ただし初期カードカタログでは、すべての攻撃カードの`effects`を空配列とする。ゲームエンジンは配置・連鎖時の効果入力、`onPlay`計画、`continuous`登録の処理経路だけを初期実装から備える。

### 4.3 サポートカード

カード固有の効果を発動するカード。

サポートカードは、少なくとも次の情報を持つ。

- カード名
- 属性
- コスト
- 効果
- 効果の発動形式
- 効果の継続期間

---

## 5. 属性

ゲームには3種類の属性が存在する。

属性の具体的な名称はカード定義で管理する。

攻撃カードおよびサポートカードは、必ず1種類の属性を持つ。

次のカードは作成しない。

- 複数属性のみなもとを同時に要求するカード
- 無属性カード
- 任意の属性のみなもとで使用できるカード

---

## 6. デッキ構築ルール

デッキは、次の条件をすべて満たさなければならない。

各カード定義は所属陣営を1つ持つ。デッキは選択した陣営と同じ陣営のカードだけで構成し、災害側カードと対策側カードを混在させてはならない。初期実装では災害側30枚と対策側30枚のスターターデッキを1種類ずつ提供する。

### 6.1 枚数条件

| カード種別     |        枚数条件 |
| -------------- | --------------: |
| デッキ合計     |    30枚ちょうど |
| みなもとカード | 8枚以上12枚以下 |
| 攻撃カード     |        11枚以上 |
| サポートカード |         7枚以下 |

### 6.2 同名カード制限

- 攻撃カードは同名2枚まで
- サポートカードは同名2枚まで
- みなもとカードには同名枚数制限を設けない

### 6.3 属性条件

3種類すべての属性をデッキに含めなければならない。

各属性について、次のカードを最低1枚ずつ含める。

- その属性のみなもとカード
- その属性の攻撃カードまたはサポートカード

デッキ構築条件は、今後のゲームバランス調整によって変更される可能性がある。

---

## 7. ゲーム開始処理

ゲーム開始前に、災害側と対策側が1人ずつであること、および各プレイヤーの30枚デッキが本人の陣営と一致することを検証する。

### 7.1 デッキのシャッフル

デッキのシャッフルはバックエンドで実行する。

フロントエンドには、シャッフル処理途中の状態を送信せず、確定したゲーム状態だけを送信する。

ゲームエンジンでは、テスト結果を再現できるように、乱数生成処理を外部から注入できる構造とする。

バックエンドは対戦作成時に非公開の乱数seedを確定し、同じゲームの初期化再実行では同じseedを使用する。seedと山札順序は対戦終了前にクライアントへ公開しない。

### 7.2 第1ラウンドの先攻

第1ラウンドの先攻プレイヤーは、バックエンド上の乱数によって決定する。

### 7.3 初期手札

各プレイヤーは、シャッフルされた山札から5枚引く。

最初に引いた5枚がすべてみなもとカードだった場合は、次の処理を行う。

1. 引いた5枚を山札に戻す
2. デッキを再度シャッフルする
3. 再び5枚引く
4. 攻撃カードまたはサポートカードが最低1枚含まれるまで繰り返す

引き直しは最大20回とする。20回目でも5枚すべてがみなもとカードの場合、ゲーム開始を失敗させ、ゲーム状態を保存しない。

引き直し対象となったみなもとカードは、みなもと総量に加算しない。

初期手札が確定した後、手札に含まれるみなもとカードを処理する。

1. 対応する属性のみなもと総量を1増加させる
2. みなもとカードを捨て札に移動する
3. 追加のカードは引かない

そのため、ゲーム開始時の実際の手札枚数は5枚未満になる場合がある。

---

## 8. ゲーム領域

各プレイヤーは、次の領域を持つ。

### 8.1 山札

まだ引いていないカードを保持する。

山札のカード内容および順序は、どちらのプレイヤーにも公開しない。

### 8.2 手札

プレイヤーが使用可能なカードを保持する。

通常の手札上限は5枚とする。

カード効果によるドローでも、手札上限を超えてカードを保持することはできない。

### 8.3 攻撃グループ領域

攻撃カードを配置する領域。

プレイヤーごとに最大5グループまで作成できる。

### 8.4 サポートゾーン

サポートカードを配置する領域。

UI上は1つのサポート枠として表示するが、内部には複数のサポートカードを配置できる。

サポートカードの配置枚数には、基本ルール上の固定上限を設けない。

### 8.5 みなもと表示領域

属性ごとのみなもと総量、使用中みなもと、使用可能みなもとを表示する領域。

みなもとカード自体は、この領域には残らない。

### 8.6 捨て札

使用済みまたは捨てられたカードを保持する。

捨て札は移動した順番を保持する。

捨て札に含まれるカードの内容および順番は、双方のプレイヤーに公開する。

---

## 9. みなもとのルール

### 9.1 みなもとの総量

みなもとの総量は、属性ごとに管理する。

みなもとカードを引くたびに、対応する属性のみなもと総量が1増加する。

### 9.2 使用中みなもと

現在バトルゾーンに存在する攻撃グループおよびサポートカードが必要とするコストの合計を、使用中みなもととする。

属性ごとに、次の式で計算する。

```text
使用可能みなもと
= みなもと総量
- 攻撃グループが必要とするコストの合計
- 場に残るサポートカードが必要とするコストの合計
```

使用可能みなもとは0未満になってはならない。

### 9.3 攻撃グループのコスト

攻撃グループのコストは、そのグループに含まれるカードのうち、最も高いコストとする。

```text
グループコスト
= グループ内の最大カードコスト
```

例：

| カード  | コスト |
| ------- | -----: |
| カードA |      2 |
| カードB |      4 |
| カードC |      3 |

このグループのコストは4となる。

### 9.4 グループ構成変更時の再計算

攻撃グループのコストは、常に現在のカード構成から再計算する。

グループコストが5から4に減少した場合は、使用中みなもとも5から4に減少し、差分の1が再び使用可能になる。

グループに現在の最大コストを超えるカードを追加した場合は、増加した差額分だけ使用中みなもとが増える。

### 9.5 みなもと減少効果

みなもとを減らすサポート効果は、対象属性のみなもと総量を永続的に減少させる。

ただし、次の条件を守る。

- みなもと総量を0にはしない
- みなもと総量を現在の使用中みなもと未満にはしない
- 指定された減少量をすべて適用できない場合は、減らせる量だけ減らす
- 既存の攻撃カードやサポートカードが、みなもと不足によって停止する状態は作らない

例：

```text
みなもと総量：5
使用中みなもと：4
指定減少量：3
```

この場合、実際に減少する量は1となる。

処理後は次の状態になる。

```text
みなもと総量：4
使用中みなもと：4
使用可能みなもと：0
```


## 11. 攻撃カード配置フェーズ

### 11.1 配置可能なプレイヤー

先攻プレイヤー配置フェーズでは、先攻プレイヤーだけが操作できる。

後攻プレイヤー配置フェーズでは、後攻プレイヤーだけが操作できる。

### 11.2 配置可能条件

攻撃カードは、次の条件をすべて満たす場合に配置できる。

- 現在のフェーズで操作権を持っている
- 対象カードが自分の手札に存在する
- 配置後も、その属性の使用中みなもとがみなもと総量を超えない
- 新規グループとして配置する場合、攻撃グループ数が5未満である
- 既存グループに連鎖する場合、連鎖条件を満たしている

### 11.3 新規グループ作成

攻撃カードを空いている攻撃グループ枠へ配置すると、新しい攻撃グループを作成する。

攻撃グループは最大5つまで作成できる。

すでに5グループ存在する場合、新しい6番目のグループは作成できない。

### 11.4 既存グループへの連鎖

すでに5グループ存在する場合でも、既存グループへの連鎖は可能とする。

連鎖されたカードは、同じ攻撃グループに追加される。

### 11.5 配置後の移動禁止

一度バトルゾーンに配置した攻撃カードは、通常操作では移動できない。

禁止される操作には、次を含む。

- 手札に戻す
- 別グループへ移動する
- グループ内の順番を変更する
- グループを分割する
- 複数グループを統合する
- 配置操作を取り消す

カード効果に移動が明記されている場合に限り、カード効果による移動を許可できる。

初期実装では、カードを移動させる効果は作成しない。

### 11.6 ラウンドをまたぐカード

攻撃カードおよび攻撃グループは、カード効果によって取り除かれない限り、ラウンド終了後もバトルゾーンに残る。

残ったカードは、次のラウンド以降も総パワーに加算される。

既存の攻撃グループには、ラウンドをまたいで新しいカードを連鎖できる。

### 11.7 攻撃配置時間

各プレイヤーの攻撃配置時間は最大90秒とする。

制限時間を超えた場合は、次の処理を行う。

- サーバー上ですでに確定している配置操作を維持する
- サーバー上ですでに確定している捨て札操作を維持する
- 未操作の手札はそのまま残す
- 自動的に配置フェーズを終了する
- クライアント上でドラッグ中など、サーバーに送信されていない操作は無効とする

## 12. 連鎖ルール

### 12.1 連鎖情報の保持

下側に配置されている攻撃カードが、その上に連鎖可能なカード情報を持つ。

### 12.2 連鎖判定

新しいカードを連鎖する際は、対象グループの一番上にあるカードとの関係だけを確認する。

グループ内のそれ以外のカードとの連鎖条件は確認しない。

### 12.3 属性制限

異なる属性の攻撃カード同士を同じグループに連鎖することはできない。

### 12.4 作成済みグループの再検証

連鎖条件は、カードを追加する瞬間だけ検証する。

作成済みグループからカードが取り除かれた場合でも、残ったカード同士の連鎖条件は再検証しない。

そのため、カード除去後に通常の連鎖条件を満たさない並びになった場合でも、そのグループはそのまま存在できる。

### 12.5 連鎖枚数

1つのグループに連鎖できるカード枚数には、ルール上の上限を設けない。

UI上は、ホバー時の詳細表示やモーダル表示などを利用して、重なったカードを確認できるようにする。

### 12.6 グループ除去

初期実装における攻撃カード除去は、グループ全体を対象とする。

グループ全体が捨て札に移動した場合は、攻撃グループ枠が1つ空く。

グループがなくなることで、それまで使用していたみなもとは即座に使用可能状態へ戻る。

---

## 13. 手札からの捨て札操作

自分の攻撃カード配置フェーズ中、手札のカードを任意の枚数だけ捨てることができる。

次のルールを適用する。

- 攻撃カードを捨てられる
- サポートカードを捨てられる
- 1ラウンドに捨てられる枚数の上限はない
- 手札のカードをすべて捨てることもできる
- 一度捨てたカードは手札に戻せない
- みなもとカードは自動処理されるため、プレイヤー自身の操作では捨てられない

---

## 14. サポートフェーズ

### 14.1 開始条件

先攻および後攻の攻撃カード配置フェーズが終了した後、サポートフェーズへ移行する。

### 14.2 操作方式

サポートフェーズ中は、双方のプレイヤーが任意のタイミングでサポートカードを使用できる。

交互に1枚ずつ使用する必要はない。

サポートカードは、バックエンドがコマンドを受信した順番で1件ずつ処理する。

### 14.3 即時解決

サポートカードの効果は、原則として即時解決する。

処理順は次のとおりとする。

1. 使用可能なフェーズか確認する
2. カードが手札に存在するか確認する
3. 対象が有効か確認する
4. みなもとが足りているか確認する
5. サポートカードをサポートゾーンへ配置する
6. 効果を解決する
7. 効果期間に応じてカードを維持または捨て札へ移動する

### 14.4 サポート終了宣言

各プレイヤーは、サポート終了ボタンを押して、自分のサポート操作を終了できる。

終了宣言は取り消せない。

一度終了宣言をしたプレイヤーは、そのサポートフェーズ中に新しいサポートカードを使用できない。

片方のプレイヤーだけが終了した場合、もう一方のプレイヤーは引き続きサポートカードを何枚でも使用できる。

次のいずれかを満たした場合、サポートフェーズを終了する。

- 双方のプレイヤーが終了宣言した
- サポートフェーズ開始から60秒が経過した

### 14.5 制限時間

サポートフェーズの制限時間は、双方で共有する60秒とする。

サーバーが制限時間内に受信したコマンドだけを有効とする。

制限時間内に受理されたコマンドは、処理中に制限時間を超えた場合でも最後まで解決する。

制限時間後に受信した新しいコマンドは拒否する。

---

## 15. サポートカードの効果形式

初期実装では、次の2種類の効果発動形式を使用する。

### 15.1 `onPlay`

サポートカードを使用したときに、一度だけ効果を発動する。

### 15.2 `continuous`

サポートカードがサポートゾーンに存在している間、継続して効果を適用する。

---

## 16. サポートカードの継続期間

サポートカードは、次のいずれかの継続期間を持つ。

```text
instant
untilRoundEnd
permanent
```

### 16.1 `instant`

使用時に効果を解決し、解決後すぐに捨て札へ移動する。

- 初期実装では、原則として`onPlay`効果として扱う
- 効果解決中だけコスト条件を確認する
- 捨て札へ移動した後は、そのカードが必要としていたみなもとが使用可能状態に戻る

### 16.2 `untilRoundEnd`

勝敗判定が終了するまでサポートゾーンに残る。

- サポートゾーンに存在している間、コスト分のみなもとを使用する
- スコア計算に効果を反映する
- 勝敗判定終了後に捨て札へ移動する
- 捨て札へ移動した後、使用していたみなもとが使用可能状態に戻る
- 効果も同時に終了する

### 16.3 `permanent`

カード効果によって場から取り除かれるまで、サポートゾーンに残り続ける。

- サポートゾーンに存在している間、コスト分のみなもとを使用する
- 場から離れた場合は捨て札へ移動する
- 場から離れた時点で効果を終了する
- 使用していたみなもとは使用可能状態に戻る

---

## 17. 不正操作とエラー処理

### 17.1 不正な対象

カード効果の対象が存在しない、または対象条件を満たさない場合は、その操作を拒否する。

操作を拒否した場合は、次の状態を維持する。

- カードは手札に残る
- みなもとは使用しない
- ゲーム状態は変更しない

### 17.2 効果処理中のエラー

効果処理中にエラーが発生した場合は、操作全体を取り消す。

一部の状態変更だけを残してはならない。

### 17.3 同時操作

複数のサポートカード使用コマンドがほぼ同時に送信された場合でも、サーバーが受信した順番で1件ずつ処理する。

先に処理された効果によって後続操作の対象が無効になった場合は、後続操作を拒否する。

コマンドは、その操作対象フェーズを一意に識別する`phaseSequence`を含む。現在のフェーズ世代と一致しないコマンドは、過去ラウンドまたは過去フェーズから遅延した操作として拒否する。

クライアントが見ていた状態バージョンが古い場合でも、それだけを理由に通常操作を拒否しない。サーバー受信順に最新状態へ適用し、カード位置、対象、みなもとなどを再検証する。制限時間にはバックエンドが記録した受信時刻を使用する。

---

## 18. 攻撃力計算

### 18.1 カード単体の攻撃力

カード単体の最終攻撃力は、次の順番で計算する。

1. 基礎攻撃力
2. 攻撃力の上書き
3. 加算・減算
4. 乗算
5. 小数点以下切り捨て
6. 最低値1を適用

複数の上書き効果が存在する場合は、後から有効になった上書き効果を採用する。

カード単体の最終攻撃力は、1未満にならない。

### 18.2 グループ攻撃力

グループ攻撃力は、次の順番で計算する。

1. グループ内の各カードについて最終攻撃力を計算する
2. 各カードの最終攻撃力を合計する
3. グループ全体を対象とする効果を適用する
4. 小数点以下を切り捨てる
5. 最低値0を適用する

グループ攻撃力は0未満にならない。

### 18.3 プレイヤー総パワー

プレイヤー総パワーは、次の順番で計算する。

1. 各攻撃グループの攻撃力を計算する
2. すべての攻撃グループの攻撃力を合計する
3. プレイヤー総パワーを対象とする効果を適用する
4. 小数点以下を切り捨てる
5. 最低値0を適用する

プレイヤー総パワーは0未満にならない。

---

## 19. スコア計算

サポートフェーズ終了後、双方の最終総パワーを自動的に計算する。

### 19.1 総パワーに差がある場合

総パワーが低いプレイヤーは、双方の総パワー差と同じ値だけスタミナを失う。

例：

```text
プレイヤーAの総パワー：12
プレイヤーBの総パワー：8
```

この場合、プレイヤーBはスタミナを4失う。

### 19.2 総パワーが同じ場合

双方の総パワーが同じ場合は、どちらのスタミナも変化しない。

### 19.3 スタミナの下限

スタミナには下限を設けない。

ダメージやカード効果によって、スタミナがマイナスの値になることを許容する。

### 19.4 勝敗判定のタイミング

サポートフェーズ中にスタミナが0以下になった場合でも、その時点ではゲームを終了しない。

スコア計算とスタミナ反映がすべて終了した後に、1度だけ勝敗を判定する。

---

## 20. 通常の勝敗判定

スコア計算終了後、次の順番で判定する。

### 20.1 片方だけスタミナが0以下

片方のスタミナだけが0以下の場合、もう片方のプレイヤーを勝者とする。

### 20.2 双方のスタミナが0以下

双方のスタミナが0以下の場合、引き分けとする。

### 20.3 双方のスタミナが1以上

ゲームを続行する。

---

## 21. 次ラウンドの先攻決定

次ラウンドの先攻は、直前のラウンドにおける最終総パワーで決定する。

### 21.1 総パワーに差がある場合

総パワーが高かったプレイヤーが、次ラウンドの先攻となる。

実際にスタミナダメージが軽減または無効化された場合でも、先攻の決定には最終総パワーを使用する。

### 21.2 総パワーが同じ場合

総パワーが同じだった場合は、直前のラウンドの先攻と後攻を入れ替える。

---

## 22. 場の整理

通常のラウンド終了時は、次の処理を行う。

- 攻撃カードおよび攻撃グループは、そのまま残す
- `permanent`サポートカードは、そのまま残す
- `untilRoundEnd`サポートカードを捨て札へ移動する
- 終了した一時効果を削除する
- 場から離れたカードが使用していたみなもとを解放する

`untilRoundEnd`効果は、スコア計算および勝敗判定が終了した後に削除する。

---

## 23. 手札補充

### 23.1 補充開始時の山札確認

手札補充フェーズ開始時に、どちらか一方または双方の山札が0枚だった場合は、カードを引かずに山札切れ判定へ進む。

手札がすでに5枚あった場合でも、山札が0枚なら山札切れとなる。

### 23.2 通常補充

山札が1枚以上ある場合、各プレイヤーは手札が5枚になるまでカードを引く。

ただし、山札の残り枚数が必要枚数より少ない場合は、引ける枚数だけ引く。

双方の補充処理は、同じ補充フェーズ内で行う。

### 23.3 みなもとカードの処理

補充によって引いたカードは、いったん手札に加える。

その後、引いたみなもとカードについて次の処理を行う。

1. 対応する属性のみなもと総量を1増加させる
2. みなもとカードを捨て札へ移動する
3. みなもとカードが抜けた分の追加補充は行わない

例：

```text
補充前の手札：3枚
2枚補充
引いたカード：
・みなもとカード1枚
・攻撃カード1枚
```

処理後は次の状態になる。

```text
みなもと総量：1増加
最終手札：4枚
追加ドロー：なし
```

### 23.4 山札が途中で空になった場合

補充処理中に1枚以上カードを引いた後で山札が0枚になった場合、そのラウンドでは山札切れとしない。

次回の手札補充フェーズ開始時に山札が0枚であれば、山札切れとなる。

---

## 24. カード効果によるドロー

カード効果によってカードを引く場合は、次のうち最も少ない枚数だけ引く。

- カード効果で指定された枚数
- 手札上限5枚までの空き枚数
- 山札の残り枚数

引いたカードがみなもとカードだった場合は、即座にみなもと総量を1増加させ、捨て札へ移動する。

みなもとカードが手札から抜けた場合でも、追加のカードは引かない。

カード効果によるドローで山札が0枚になっても、即座には山札切れとしない。

山札切れは、手札補充フェーズ開始時に判定する。

---

## 25. 山札切れ

手札補充フェーズ開始時に、どちらか一方または双方の山札が0枚だった場合、山札切れとしてゲームを終了する。

山札切れ時は、双方の現在スタミナを比較する。

- スタミナが高いプレイヤーの勝利
- スタミナが同じ場合は引き分け

山札切れ時には、総パワーによる追加判定を行わない。

---

## 26. 最大ラウンド数

ゲームは最大30ラウンドとする。

第30ラウンドは、次の順番で処理する。

1. サポートフェーズ終了
2. スコア計算
3. スタミナ反映
4. 通常のスタミナ勝敗判定
5. 決着していなければ最大ラウンド判定
6. スタミナ比較
7. スタミナが同じなら総パワー比較
8. 総パワーも同じなら引き分け

第30ラウンド終了時に通常のスタミナ勝敗が成立した場合は、その結果を優先する。

通常の勝敗が成立しなかった場合は、次の順番で勝敗を決める。

1. スタミナが高いプレイヤー
2. スタミナが同じ場合、最終総パワーが高いプレイヤー
3. スタミナと最終総パワーが同じ場合、引き分け

第30ラウンド終了後は、場の整理および手札補充フェーズには進まない。

---

## 27. 勝敗条件の優先順位

勝敗条件は、次の優先順位で適用する。

1. スコア計算後のスタミナ勝敗
2. 第30ラウンド終了時の最大ラウンド判定
3. 手札補充フェーズ開始時の山札切れ判定

---

## 29. 切断と再接続

プレイヤーが通信切断した場合でも、ゲーム内のフェーズタイマーは停止しない。

- 切断中も攻撃配置時間は進行する
- 切断中もサポート時間は進行する
- 時間切れ処理は通常どおり実行する
- プレイヤーは再接続できる
- 再接続時は、バックエンド上の最新ゲーム状態を取得する

---

## 30. ゲームロジックとUI演出

ゲームロジックの処理と、UIアニメーションの処理は分離する。

バックエンドは、UIアニメーションの完了を待たずにゲーム結果を確定する。

バックエンドが確定する処理には、次を含む。

- 総パワー計算
- スタミナ変更
- 勝敗判定
- 場の整理
- 手札補充
- みなもとカード処理
- 次ラウンド開始

フロントエンドは、バックエンドから受け取った確定結果およびゲームイベントを利用して演出を行う。

演出例：

```text
サポートフェーズ終了
↓
「総パワー計算」の表示
↓
総パワーのカウントアップ
↓
スタミナ減少アニメーション
↓
勝敗判定表示
↓
手札補充アニメーション
↓
次ラウンド開始表示
```

UIアニメーションによって、バックエンドのゲーム状態を巻き戻したり、処理を遅延させたりしてはならない。

---

## 31. 基本ルールの変更管理

本書で定義した数値や制限は、ゲームバランス調整によって変更される可能性がある。

変更される可能性が高い項目には、次を含む。

- 攻撃配置時間
- サポート時間
- 最大ラウンド数
- デッキ構築条件
- カード種別ごとの枚数制限
- 攻撃グループ数
- 初期スタミナ

ルールを変更する場合は、本書のバージョンを更新し、ゲームエンジン、カード定義、テストコードとの整合性を確認する。

進行中ゲームには、開始時に使用した次のバージョンを固定する。

- `RulesetVersion`
- `CardCatalogVersion`
- `EngineSemanticsVersion`

同じバージョン識別子の内容を上書きしてはならない。進行中ゲームの再接続・再開・検証に必要なバージョンの実体を取得できるよう保持する。保持できない運用では、対戦開始時に必要なルールとカード定義のスナップショットを保存する。
